## CB-001: COMBOBOX created via MDL fails MxBuild with "definition has changed"

**Severity:** High
**Area:** MDL / Pluggable Widgets
**Status:** Open

When a `COMBOBOX` widget is created via MDL, the `.mpr` file records it with a stale
pluggable-widget definition version. MxBuild detects the mismatch and fails with a hard
error. The widget cannot be deployed until it is "updated" (which requires Studio Pro).

**Repro:**
```mdl
CREATE PAGE Summit.TestPage
(
  Params: { $Customer: Summit.Customer },
  Layout: Atlas_Core.PopupLayout
)
{
  LAYOUTGRID g {
    ROW r {
      COLUMN c (DesktopWidth: AutoFill) {
        DATAVIEW dv (DataSource: $Customer) {
          COMBOBOX cbCreditRating (Label: 'Credit Rating', Binds: CreditRating)
        }
      }
    }
  }
}
/
```

**MxBuild error:**
```
ERROR at Summit, Page 'TestPage', Combo box 'cbCreditRating':
The definition of this widget has changed. Update this widget by right-clicking it
and selecting 'Update widget', or select 'Update all widgets' to update all widgets
in the app.
Error: mxbuild failed: exit status 3
```

**Root cause:** mxcli creates the COMBOBOX widget instance using an older widget definition
than the version currently installed (DataWidgets v3.5.0). Existing comboboxes created via
Studio Pro (e.g., in `Administration.Account_New`) build successfully â€” confirming the issue
is specific to mxcli-created instances.

**Workaround:** Use `RADIOBUTTONS` instead of `COMBOBOX` for enumeration attributes.
Both render an enumeration selector; `RADIOBUTTONS` is created with a current definition
and builds without error.

```mdl
-- Use this instead:
RADIOBUTTONS rbCreditRating (Label: 'Credit Rating', Binds: CreditRating)
```

**Expected:** mxcli should create pluggable widget instances using the currently installed
widget definition version, matching what Studio Pro would produce.

## CB-002: COMBOBOX `Label` property is silently dropped on creation

**Severity:** Medium
**Area:** MDL / COMBOBOX
**Status:** Open

When a `COMBOBOX` is created via MDL with a `Label` property, the label is accepted
without error but is not persisted. `DESCRIBE PAGE` shows the widget without a `Label`.
This affects all comboboxes â€” including those in marketplace modules (e.g., `Administration.Account_New`
shows `COMBOBOX comboBox1 (Binds: System.UserRole.Name)` with no label).

**Repro:**
```mdl
COMBOBOX cbCreditRating (Label: 'Credit Rating', Binds: CreditRating)
```

**DESCRIBE output:**
```mdl
COMBOBOX cbCreditRating (Binds: Summit.Customer.CreditRating)
-- Label: 'Credit Rating' is missing
```

**Note:** This may be a DESCRIBE output issue rather than a storage issue â€” the label may be
stored internally as a pluggable widget property that DESCRIBE does not know how to render.
However, since CHECKBOX and TEXTBOX both preserve and display their `Label` property
correctly, the inconsistency warrants investigation.

**Workaround:** None needed for RADIOBUTTONS (which is the recommended replacement for
COMBOBOX â€” see CB-001). `RADIOBUTTONS` preserves its `Label` correctly.
