# mxcli: AI-Powered Development for Mendix

**Accelerate Mendix Development with Natural Language and Code Generation**

## Executive Summary

**mxcli** is a command-line interface that brings AI-powered development to the Mendix platform. By providing a SQL-like language called **MDL (Mendix Definition Language)**, mxcli enables developers and AI assistants to read, understand, and modify Mendix projects programmatically.

### Key Benefits

- **10x Faster Development**: Generate domain models, microflows, and pages using natural language
- **Consistent Standards**: Define organizational best practices as MDL templates
- **AI-Native Integration**: Purpose-built for GenAI code assistants like Claude Code
- **Full Project Understanding**: Agentic search capabilities for intelligent code navigation
- **Token-Efficient**: MDL uses 5-10x fewer tokens than JSON model representations, reducing API costs
- **Better Code Reviews**: Higher information density makes diffs easier to read and review

## How It Works

```
┌─────────────────┐     ┌──────────────┐     ┌─────────────────┐
│   Developer     │     │    mxcli     │     │ Mendix Project  │
│   or AI Agent   │────▶│  (MDL REPL)  │────▶│   (.mpr file)   │
│                 │ MDL │              │     │                 │
│ "Create a       │     │ Parses MDL   │     │ Creates actual  │
│  Customer       │     │ Validates    │     │ entities,       │
│  entity..."     │     │ Executes     │     │ microflows,     │
└─────────────────┘     └──────────────┘     │ pages, etc.     │
                                             └─────────────────┘
```

### The MDL Advantage

MDL is a **text-based, SQL-like language** that describes Mendix model elements. Compared to the visual Studio Pro interface and JSON model representations, MDL offers:

- **Generated by AI**: LLMs produce syntactically correct MDL with high accuracy
- **Human-Readable Diffs**: Text-based format is easier to review than binary or JSON changes
- **High Information Density**: Scan complete microflow logic at a glance, without clicking through activities
- **Token Efficient**: 5-10x fewer tokens than JSON representations, allowing more application context in LLM prompts
- **Concise Skill Documents**: Smaller, more precise AI instruction files lead to better generation quality
- **Templated for Reuse**: Standard patterns become simple copy-paste snippets

### Why Token Efficiency Matters

When working with GenAI APIs, **context window size is a critical constraint**. MDL's compact syntax provides significant advantages over JSON model representations:

| Representation | Tokens for a 10-entity module |
|---------------|------------------------------|
| JSON (raw model) | ~15,000-25,000 tokens |
| MDL | ~2,000-4,000 tokens |
| **Savings** | **5-10x fewer tokens** |

**Benefits of token efficiency:**

1. **Lower API costs**: Fewer tokens = lower costs per request
2. **More context available**: Fit more of your application in a single prompt
3. **Better AI understanding**: Less noise, more signal in the context
4. **Faster responses**: Smaller payloads process more quickly
5. **Larger scope operations**: Modify multiple entities, microflows, and pages in one request

**Example comparison:**

```json
// JSON representation (~180 tokens)
{
  "entity": {
    "name": "Customer",
    "documentation": "Customer master data",
    "attributes": [
      {"name": "Name", "type": {"type": "String", "length": 200}},
      {"name": "Email", "type": {"type": "String", "length": 200}}
    ]
  }
}
```

```sql
-- MDL representation (~35 tokens)
/** Customer master data */
CREATE PERSISTENT ENTITY Sales.Customer (
    Name: String(200),
    Email: String(200)
);
```

This efficiency extends to skill documents—the AI instruction files that guide generation. Concise MDL examples in skills mean:
- More patterns fit in the skill document
- AI has clearer examples to follow
- Generated output is more accurate and consistent

## Integration with GenAI Tools

### Multi-Tool AI Assistant Support

mxcli is designed from the ground up to work with AI coding assistants. It supports **five AI tools** out of the box:

| Tool | Flag | What's Created |
|------|------|----------------|
| **Claude Code** (default) | `--tool claude` | `.claude/settings.json`, `CLAUDE.md`, commands, lint rules |
| **Cursor** | `--tool cursor` | `.cursorrules` |
| **Continue.dev** | `--tool continue` | `.continue/config.json` |
| **Windsurf** | `--tool windsurf` | `.windsurfrules` |
| **Aider** | `--tool aider` | `.aider.conf.yml` |

```bash
# Initialize with Claude Code (default)
mxcli init

# Initialize for Cursor
mxcli init --tool cursor

# Initialize for multiple tools
mxcli init --tool claude --tool cursor --tool continue

# Initialize for all supported tools
mxcli init --all-tools

# List supported tools
mxcli init --list-tools
```

When you run `mxcli init`, it creates a **universal** structure shared by all tools, plus tool-specific configuration:

The AI assistant can then:

1. **Understand your project** by querying the model
2. **Generate MDL** based on natural language requests
3. **Validate changes** before applying them

### Example: Natural Language to Working Feature

**Developer says**: *"Create a customer management module with name, email, and phone fields. Add a microflow to validate the email format."*

**Claude Code generates**:

```sql
/** Create the customer management module */
CREATE MODULE CustomerManagement;

/**
 * Customer entity with contact information.
 * Master data for CRM functionality.
 */
@Position(100, 100)
CREATE PERSISTENT ENTITY CustomerManagement.Customer (
    Name: String(200) NOT NULL,
    Email: String(200) NOT NULL UNIQUE,
    Phone: String(50),
    IsActive: Boolean DEFAULT true,
    CreatedAt: DateTime DEFAULT '[%CurrentDateTime%]'
);

/**
 * Validate email format.
 * Returns true if email contains @ and . characters.
 */
CREATE MICROFLOW CustomerManagement.ValidateEmail(
    $email: String
) RETURNS Boolean AS $isValid
BEGIN
    DECLARE $isValid Boolean = false;

    IF $email != empty AND contains($email, '@') AND contains($email, '.') THEN
        SET $isValid = true;
    END IF;

    RETURN $isValid;
END;
```

**Result**: A fully functional module appears in Studio Pro, ready for use.

### Modules and Entities

The `@Position(x, y)` annotation controls where entities appear in the Studio Pro domain model diagram. Coordinates are in pixels from the top-left. This helps maintain clean, organized visual layouts when generating multiple entities.

```sql
/** Create a new module for sales functionality */
CREATE MODULE Sales;

/**
 * Order entity with business rules and audit support.
 * Sales orders with full audit trail.
 */
@Position(100, 100)
CREATE PERSISTENT ENTITY Sales.Order (
    /** Unique order identifier for external systems */
    OrderNumber: String(50) NOT NULL UNIQUE,

    /** Order timestamp, defaults to creation time */
    OrderDate: DateTime DEFAULT '[%CurrentDateTime%]',

    /** Monetary value with precision */
    TotalAmount: Decimal NOT NULL DEFAULT 0,

    /** Order lifecycle status */
    Status: Enumeration(Sales.OrderStatus) DEFAULT 'Draft',

    /** Soft delete support */
    IsArchived: Boolean DEFAULT false
);

/**
 * Order status enumeration for lifecycle tracking.
 * Lifecycle states for sales orders.
 */
CREATE ENUMERATION Sales.OrderStatus (
    Draft 'Draft Order',
    Submitted 'Submitted for Processing',
    Approved 'Approved',
    Shipped 'Shipped to Customer',
    Completed 'Order Completed',
    Cancelled 'Order Cancelled'
);

/**
 * Link orders to customers.
 * Each order belongs to exactly one customer.
 */
CREATE ASSOCIATION Sales.Order_Customer
    FROM Sales.Order
    TO Sales.Customer
    TYPE Reference
    OWNER Default
    DELETE_BEHAVIOR DELETE_BUT_KEEP_REFERENCES;
```

### Microflows (Business Logic)

```sql
/**
 * Process a new order submission
 * Validates the order, updates inventory, and sends confirmation
 */
CREATE MICROFLOW Sales.SubmitOrder(
    $order: Sales.Order
) RETURNS Boolean AS $success
FOLDER 'Order Processing'
BEGIN
    DECLARE $success Boolean = false;

    /** Validate order has items */
    RETRIEVE $orderLines FROM Sales.OrderLine
        WHERE Sales.OrderLine_Order = $order;

    IF length($orderLines) = 0 THEN
        LOG WARNING NODE 'OrderValidation' 'Order has no items, cannot submit';
        RETURN false;
    END IF;

    /** Calculate total from line items */
    DECLARE $total Decimal = 0;
    LOOP $line IN $orderLines
    BEGIN
        SET $total = $total + $line/LineTotal;
    END LOOP;

    /** Update order status and commit */
    CHANGE $order (
        TotalAmount = $total,
        Status = 'Submitted',
        SubmittedAt = [%CurrentDateTime%]);
    COMMIT $order WITH EVENTS;

    /** Send confirmation email */
    CALL MICROFLOW Sales.SendOrderConfirmation(order = $order);

    SET $success = true;
    RETURN $success;
END;

/**
 * Retrieve orders for dashboard display
 * Returns recent orders with customer information
 */
CREATE MICROFLOW Sales.GetRecentOrders()
RETURNS List of Sales.Order AS $orders
BEGIN
    RETRIEVE $orders FROM (
        SELECT o.*
        FROM Sales.Order AS o
        WHERE o/IsArchived = false
        ORDER BY o/OrderDate DESC
        LIMIT 50
    );

    RETURN $orders;
END;
```

### Security Management

```sql
/**
 * Module roles define permissions within a module.
 * Create roles for different access levels.
 */
CREATE MODULE ROLE Shop.Admin DESCRIPTION 'Full administrative access';
CREATE MODULE ROLE Shop.User DESCRIPTION 'Regular user access';
CREATE MODULE ROLE Shop.Viewer DESCRIPTION 'Read-only access';

/**
 * Grant microflow execute access to roles.
 */
GRANT EXECUTE ON MICROFLOW Shop.ACT_Customer_Create TO Shop.User, Shop.Admin;
GRANT EXECUTE ON MICROFLOW Shop.ACT_Customer_Delete TO Shop.Admin;

/**
 * Grant page view access to roles.
 */
GRANT VIEW ON PAGE Shop.Customer_Overview TO Shop.User, Shop.Admin, Shop.Viewer;
GRANT VIEW ON PAGE Shop.Customer_Edit TO Shop.User, Shop.Admin;

/**
 * Entity access rules with CRUD permissions.
 * Supports selective member access and XPath constraints.
 */
GRANT Shop.Admin ON Shop.Customer (CREATE, DELETE, READ *, WRITE *);
GRANT Shop.User ON Shop.Customer (READ (Name, Email), WRITE (Email));
GRANT Shop.Viewer ON Shop.Customer (READ *);
GRANT Shop.User ON Shop.Order (READ *, WRITE *) WHERE '[Status = ''Open'']';

/**
 * User roles aggregate module roles from multiple modules.
 */
CREATE USER ROLE AppAdmin (Shop.Admin) MANAGE ALL ROLES;
CREATE USER ROLE AppUser (Shop.User);

/**
 * Project security settings.
 */
ALTER PROJECT SECURITY LEVEL PRODUCTION;
ALTER PROJECT SECURITY DEMO USERS ON;

/**
 * Demo users for development/testing.
 */
CREATE DEMO USER 'demo_admin' PASSWORD 'Admin123!' (AppAdmin);

/**
 * Inspect security with SHOW commands.
 */
SHOW PROJECT SECURITY;
SHOW MODULE ROLES IN Shop;
SHOW ACCESS ON MICROFLOW Shop.ACT_Customer_Create;
SHOW SECURITY MATRIX IN Shop;
```

## AI Generation Skills: Combining Capabilities

mxcli includes specialized **skills** that guide AI assistants in generating complete, production-ready features. These skills can be combined to create entire application modules from a single request.

### Available Generation Skills

| Skill | Purpose | Output |
|-------|---------|--------|
| **Domain Model** | Generate entities with relationships | Entities, associations, enumerations |
| **Overview Pages** | Create list/grid pages for entities | Data grids with search, sort, CRUD buttons |
| **Detail Pages** | Generate edit/view forms | Data views with input widgets |
| **Validation Microflows** | Add business rule validation | Microflows with feedback messages |
| **CRUD Microflows** | Generate create/update/delete logic | Standard data operations |
| **REST Clients** | Integrate external APIs | REST client definitions with operations |
| **OQL Queries** | Build complex data retrieval | Optimized queries with joins |
| **Security** | Configure access control | Module roles, user roles, GRANT/REVOKE |

### Skill Combinations

**Example 1: Complete Entity Module**

*Request*: "Create a Product catalog with name, price, category, and inventory tracking"

*AI combines*: Domain Model + Overview Page + Detail Page + Validation

```sql
/**
 * === DOMAIN MODEL ===
 * Generated by: Domain Model Skill
 */
CREATE MODULE ProductCatalog;

/** Product categories for catalog organization */
CREATE ENUMERATION ProductCatalog.ProductCategory (
    Electronics 'Electronics',
    Clothing 'Clothing',
    HomeGarden 'Home & Garden',
    Sports 'Sports & Outdoors',
    Books 'Books & Media'
);

/**
 * Product catalog with inventory tracking.
 * Central product master data for e-commerce operations.
 */
@Position(100, 100)
CREATE PERSISTENT ENTITY ProductCatalog.Product (
    /** Product identification */
    SKU: String(50) NOT NULL UNIQUE,
    Name: String(200) NOT NULL,
    Description: String(2000),

    /** Pricing */
    Price: Decimal NOT NULL,
    CostPrice: Decimal,

    /** Categorization */
    Category: Enumeration(ProductCatalog.ProductCategory) NOT NULL,

    /** Inventory */
    StockQuantity: Integer DEFAULT 0,
    ReorderLevel: Integer DEFAULT 10,
    IsActive: Boolean DEFAULT true,

    /** Audit fields */
    CreatedAt: DateTime DEFAULT '[%CurrentDateTime%]',
    ModifiedAt: DateTime
);

-- Note: Indexes are part of entity definitions, or added via ALTER ENTITY:
-- ALTER ENTITY ProductCatalog.Product ADD INDEX (Category, IsActive);

/**
 * === VALIDATION MICROFLOWS ===
 * Generated by: Validation Skill
 */
CREATE MICROFLOW ProductCatalog.ValidateProduct($product: ProductCatalog.Product)
RETURNS Boolean AS $isValid
FOLDER 'Validation'
BEGIN
    DECLARE $isValid Boolean = true;

    /** SKU format validation */
    IF length($product/SKU) < 3 THEN
        VALIDATION FEEDBACK $product/SKU MESSAGE 'SKU must be at least 3 characters';
        SET $isValid = false;
    END IF;

    /** Price validation */
    IF $product/Price <= 0 THEN
        VALIDATION FEEDBACK $product/Price MESSAGE 'Price must be greater than zero';
        SET $isValid = false;
    END IF;

    /** Margin validation */
    IF $product/CostPrice != empty AND $product/CostPrice >= $product/Price THEN
        VALIDATION FEEDBACK $product/CostPrice MESSAGE 'Cost price should be less than selling price';
        SET $isValid = false;
    END IF;

    /** Inventory validation */
    IF $product/ReorderLevel < 0 THEN
        VALIDATION FEEDBACK $product/ReorderLevel MESSAGE 'Reorder level cannot be negative';
        SET $isValid = false;
    END IF;

    RETURN $isValid;
END;

/**
 * === OVERVIEW PAGE ===
 * Generated by: Overview Page Skill
 */
CREATE PAGE ProductCatalog.ProductOverview
(
  Title: 'Product Catalog',
  Layout: Atlas_Core.Atlas_Default,
  Url: 'products'
)
{
  LAYOUTGRID mainGrid {
    ROW row1 {
      COLUMN col1 (DesktopWidth: 12) {
        DYNAMICTEXT pageTitle (Content: 'Product Catalog', RenderMode: H3)
```

```sql
    /** User who created this record */
    CreatedBy: String(200),

    /** Last modification timestamp */
    ModifiedAt: DateTime,
```

### Advanced Catalog Queries

#### Duplicate Entity Detection

```sql
/**
 * Find entities with similar names across modules
 * Helps identify potential consolidation opportunities
 */
SELECT
    e1.ModuleName AS Module1,
    e1.Name AS Entity1,
    e2.ModuleName AS Module2,
    e2.Name AS Entity2
FROM CATALOG.ENTITIES e1
JOIN CATALOG.ENTITIES e2
    ON LOWER(e1.Name) = LOWER(e2.Name)
    AND e1.ModuleName < e2.ModuleName
ORDER BY e1.Name;

/**
 * Find entities with identical attribute patterns
 * Detects copy-paste inheritance anti-patterns
 */
SELECT
    e1.QualifiedName AS Entity1,
    e2.QualifiedName AS Entity2,
    COUNT(*) AS MatchingAttributes
FROM CATALOG.ATTRIBUTES a1
JOIN CATALOG.ATTRIBUTES a2
    ON a1.Name = a2.Name
    AND a1.AttributeType = a2.AttributeType
    AND a1.EntityId != a2.EntityId
JOIN CATALOG.ENTITIES e1 ON a1.EntityId = e1.Id
JOIN CATALOG.ENTITIES e2 ON a2.EntityId = e2.Id
WHERE e1.QualifiedName < e2.QualifiedName
GROUP BY e1.QualifiedName, e2.QualifiedName
HAVING COUNT(*) >= 3
ORDER BY MatchingAttributes DESC;

/**
 * Find potential enum candidates
 * Entities with only String attributes and few instances
 */
SELECT
    e.QualifiedName,
    COUNT(a.Id) AS AttributeCount
FROM CATALOG.ENTITIES e
JOIN CATALOG.ATTRIBUTES a ON e.Id = a.EntityId
WHERE a.AttributeType = 'String'
GROUP BY e.QualifiedName
HAVING COUNT(a.Id) <= 2
   AND COUNT(a.Id) = (
       SELECT COUNT(*) FROM CATALOG.ATTRIBUTES
       WHERE EntityId = e.Id
   );
```

#### Data Lineage Analysis

```sql
/**
 * Entity dependency graph via associations
 * Find all entities connected to a given entity
 */
SELECT
    'Outgoing' AS Direction,
    a.Name AS Association,

FROM CATALOG.ASSOCIATIONS a
WHERE a.ParentEntity = 'Sales.Order'

FROM CATALOG.ASSOCIATIONS a

WHERE m.ObjectUsage LIKE '%Sales.Customer%'
   OR m.Parameters LIKE '%Sales.Customer%'
   OR m.ReturnType LIKE '%Sales.Customer%';

/**

 */
SELECT
    p.QualifiedName AS Page,
    p.DataSource AS Entity,
    p.WidgetTypes
FROM CATALOG.PAGES p
WHERE p.DataSource IS NOT NULL
ORDER BY p.DataSource, p.QualifiedName;

/**

 */
SELECT 'Page' AS Source, p.QualifiedName AS Element, 'displays' AS Action
FROM CATALOG.PAGES p
```

#### Quality Analysis Queries

```sql
/**
 * Entities without documentation
 * Identify technical debt in documentation
 */
SELECT
    QualifiedName,
    EntityType,
    AttributeCount
FROM CATALOG.ENTITIES
WHERE (Description IS NULL OR Description = '')
ORDER BY AttributeCount DESC;

/**
 * Documentation coverage by module
 */
SELECT
    ModuleName,
    COUNT(*) AS TotalEntities,
    SUM(CASE WHEN Description != '' THEN 1 ELSE 0 END) AS Documented,
    ROUND(100.0 * SUM(CASE WHEN Description != '' THEN 1 ELSE 0 END) / COUNT(*), 1) AS CoveragePercent
FROM CATALOG.ENTITIES
GROUP BY ModuleName
ORDER BY CoveragePercent ASC;

/**
 * Complex entities (many attributes/associations)
 * Candidates for refactoring
 */
SELECT
    e.QualifiedName,
    e.AttributeCount,
    COUNT(a.Id) AS AssociationCount,
    e.AttributeCount + COUNT(a.Id) AS ComplexityScore
FROM CATALOG.ENTITIES e
LEFT JOIN CATALOG.ASSOCIATIONS a
    ON e.QualifiedName = a.ParentEntity
    OR e.QualifiedName = a.ChildEntity
GROUP BY e.QualifiedName, e.AttributeCount
HAVING ComplexityScore > 15
ORDER BY ComplexityScore DESC;

/**
 * Orphan detection - entities with no associations
 * May indicate missing relationships
 */
SELECT e.QualifiedName, e.EntityType, e.AttributeCount
FROM CATALOG.ENTITIES e
LEFT JOIN CATALOG.ASSOCIATIONS a
    ON e.QualifiedName = a.ParentEntity
    OR e.QualifiedName = a.ChildEntity
WHERE a.Id IS NULL
  AND e.EntityType = 'Persistent'
ORDER BY e.ModuleName, e.Name;

/**
 * Naming convention violations
 * Find entities not following PascalCase
 */
SELECT QualifiedName, Name
FROM CATALOG.ENTITIES
WHERE Name != UPPER(SUBSTR(Name, 1, 1)) || SUBSTR(Name, 2)
   OR Name LIKE '%_%'
   OR Name LIKE '% %';

/**
 * Module health dashboard
 * Overview of module complexity and documentation
 */
SELECT
    ModuleName,
    COUNT(DISTINCT e.Id) AS Entities,
    COUNT(DISTINCT m.Id) AS Microflows,
    COUNT(DISTINCT p.Id) AS Pages,
    ROUND(100.0 * SUM(CASE WHEN e.Description != '' THEN 1 ELSE 0 END) /
        NULLIF(COUNT(DISTINCT e.Id), 0), 1) AS DocCoverage
FROM CATALOG.ENTITIES e
LEFT JOIN CATALOG.MICROFLOWS m ON e.ModuleName = m.ModuleName
LEFT JOIN CATALOG.PAGES p ON e.ModuleName = p.ModuleName
GROUP BY ModuleName
ORDER BY Entities DESC;
```

## External SQL Connectivity

mxcli can connect to external databases (PostgreSQL, Oracle, SQL Server) for querying, data import, and Database Connector generation. Credentials are isolated — DSN never appears in session output or logs.

### Connecting and Querying

```sql
-- Connect to an external database
SQL CONNECT postgres 'postgres://user:pass@localhost:5432/mydb' AS source;

-- Explore schema
SQL source SHOW TABLES;
SQL source SHOW VIEWS;
SQL source DESCRIBE employees;

-- Run arbitrary SQL queries
SQL source SELECT * FROM employees WHERE active = true LIMIT 10;

-- Manage connections
SQL CONNECTIONS;        -- List active connections (alias + driver only)
SQL DISCONNECT source;
```

Supported drivers: `postgres` (pg, postgresql), `oracle` (ora), `sqlserver` (mssql).

### Data Import

Import external data directly into a Mendix application's PostgreSQL database:

```sql
-- Basic import: map source columns to entity attributes
IMPORT FROM source QUERY 'SELECT name, email FROM employees'
  INTO HR.Employee
  MAP (name AS Name, email AS Email);

-- Import with association linking
IMPORT FROM source QUERY 'SELECT name, dept_name FROM employees'
  INTO HR.Employee
  MAP (name AS Name)
  LINK (dept_name TO Employee_Department ON Name);

-- Batch and limit control
IMPORT FROM source QUERY 'SELECT * FROM products'
  INTO Catalog.Product
  MAP (name AS Name, price AS Price)
  BATCH 500
  LIMIT 1000;
```

### Database Connector Generation

Auto-generate Mendix Database Connector MDL from an external database schema:

```sql
-- Generate connector for all tables
SQL source GENERATE CONNECTOR INTO HRModule;

-- Generate for specific tables, execute immediately
SQL source GENERATE CONNECTOR INTO HRModule
  TABLES (employees, departments)
  EXEC;
```

### CLI Subcommand

```bash
# Direct SQL query from command line
mxcli sql --driver postgres --dsn 'postgres://...' "SELECT 1"
```

#### Features

**Syntax and Validation:**
- **Syntax Highlighting** - Color-coded MDL keywords, types, strings, comments (TextMate grammar)
- **Parse Diagnostics** - Real-time error detection as you type (no project required)
- **Semantic Diagnostics** - Reference validation on save (checks entity/microflow/page references against `.mpr`)
- **Document Symbols** - Outline view listing all CREATE/ALTER/DROP statements
- **Folding Ranges** - Collapsible BEGIN/END, IF/END IF, LOOP/END LOOP, braces, parentheses

**Navigation:**
- **Hover** - Shows MDL definition for `Module.Entity` qualified names (via `mxcli describe`)
- **Go to Definition** - Ctrl+click/Cmd+click on qualified names opens virtual MDL document
- **Code Completion** - Context-aware keyword suggestions (e.g., after `CREATE`, `SHOW`)

**Execution:**
- **Context Menu Commands:**
  - **Run File** - Execute entire MDL file against project
  - **Run Selection** - Execute selected MDL statements
  - **Check File** - Validate syntax and references without execution

When `mdl.mprPath` is not set, the extension searches the workspace for `.mpr` files. If exactly one is found, it's used automatically.

**Project-Aware Features:**
Features requiring project access (hover, definition, semantic diagnostics) gracefully degrade when no `.mpr` is configured. Core features (syntax highlighting, parse diagnostics, completion, symbols, folding) work without a project.

#### Language Server Protocol (LSP)

The extension communicates with `mxcli lsp --stdio` as the language server. This architecture enables:
- Use with any LSP-compatible editor (not just VS Code)
- Consistent behavior across editors
- Single implementation for all MDL tooling

**LSP Server Capabilities:**
```
┌─────────────────────────────────────────────────────────────┐
│  VS Code / Cursor / Other LSP-Compatible Editors           │
├─────────────────────────────────────────────────────────────┤
│  your-script.mdl                                            │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ CREATE PERSISTENT ENTITY Sales.█                       ││
│  │                          ┌──────────────────┐          ││
│  │                          │ Customer         │          ││
│  │                          │ Order            │          ││
│  │                          │ Product          │          ││
│  │                          │ OrderLine        │          ││
│  │                          └──────────────────┘          ││
│  │                          ▲ Autocomplete suggestions    ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

**Using LSP with Other Editors:**
```bash
# Start language server manually (reads from stdin, writes to stdout)
mxcli lsp --stdio

# Configure your editor to spawn this command when opening .mdl files
```

### Code Search and Navigation

These commands require a catalog refresh first (`REFRESH CATALOG FULL` or `mxcli -p app.mpr -c "REFRESH CATALOG FULL"`):

```bash
# Find what calls a microflow (callers)
mxcli callers -p app.mpr Module.MyMicroflow
mxcli callers -p app.mpr Module.MyMicroflow --transitive

# Find what a microflow calls (callees)
mxcli callees -p app.mpr Module.MyMicroflow

# Find all references to an element
mxcli refs -p app.mpr Module.Customer

# Analyze impact of changing an element
mxcli impact -p app.mpr Module.Customer

# Assemble context for LLM consumption
mxcli context -p app.mpr Module.MyMicroflow --depth 3

# Full-text search across strings and source
mxcli search -p app.mpr "validation"
mxcli search -p app.mpr "validation" -q --format names
mxcli search -p app.mpr "validation" -q --format json
```

**Search Output Formats:**
- `names` (default) - `type<TAB>name` per line, suitable for piping
- `json` - JSON array of results
- `-q` flag - Quiet mode (no headers/formatting)

**Example: Pipe search results to describe:**
```bash
mxcli search -p app.mpr "error" -q --format names | head -1 | awk '{print $2}' | \
  xargs mxcli describe -p app.mpr microflow
```

### Linting

```bash
# Lint a project for issues
mxcli lint -p app.mpr

# With colored output
mxcli lint -p app.mpr --color

# JSON output for CI integration
mxcli lint -p app.mpr --format json

# SARIF output for GitHub integration
mxcli lint -p app.mpr --format sarif > results.sarif

# List available rules
mxcli lint -p app.mpr --list-rules

# Exclude modules from linting
mxcli lint -p app.mpr --exclude System --exclude Administration
```

**Built-in Lint Rules (Go):**
- **MDL001-MDL007** - Naming conventions, empty microflows, domain model size, validation feedback, image source, empty containers, page navigation security
- **SEC001-SEC003** - Entity access rules, password policy, demo users
- **CONV011-CONV014** - No commit in loop, exclusive split captions, error handling on external calls, no continue error handling

**Bundled Starlark Rules (.claude/lint-rules/):**
- **SEC004-SEC009** - Guest access, strict mode, PII exposure, anonymous access, member restrictions
- **ARCH001-ARCH003** - Cross-module data, microflow-based writes, entity business keys
- **QUAL001-QUAL004** - McCabe complexity, documentation, long microflows, orphaned elements
- **DESIGN001** - Entity attribute count
- **CONV001-CONV010, CONV015-CONV017** - Best practice conventions (boolean naming, page suffixes, enumeration/snippet prefixes, access rule constraints, role mapping, microflow size/content, validation rules, event handlers, calculated attributes)

Use `mxcli lint -p app.mpr --list-rules` to see all available rules.

### Catalog Management

The catalog is a SQLite database that caches project metadata for fast querying:

```bash
# Refresh catalog (within MDL commands)
mxcli -p app.mpr -c "REFRESH CATALOG"           # Quick refresh
mxcli -p app.mpr -c "REFRESH CATALOG FULL"      # Full rebuild with refs

# Query catalog tables
```

Catalog is cached in `.mxcli/catalog.db` next to the MPR file.
